# Three Scenarios: Greenfield, Brownfield, Ambiguous

Each scenario was driven **through** the orchestration engine described in
[`ORCHESTRATION_ARCHITECTURE.md`](./ORCHESTRATION_ARCHITECTURE.md) against this
repo's real codebase — the audit logs referenced below are real, generated by
actually running `python orchestrator/run.py start ...`, not hand-written examples.
Run them yourself per [`SETUP.md`](./SETUP.md).

## 1. Greenfield — QR Code Generation

**Input**: "Add QR code generation for short URLs" (well-defined, no ambiguity to resolve).

**Decomposition** (`orchestrator/artifacts/greenfield/requirements.md`,
`design.md`): normalized into an acceptance-criteria list (PNG endpoint, 404 for
unknown codes, no expiry check, decodes back to the exact `shortUrl`), then a design
identifying ZXing as the QR-encoding library (confirmed with the user before adding a
new dependency — hand-rolling QR/Reed-Solomon encoding was explicitly rejected as
low-quality/high-risk for a "production-quality" deliverable), a new `QrCodeService`,
and a new `GET /api/urls/{shortCode}/qr` endpoint.

**Orchestration** (`orchestrator/pipelines/greenfield.json`): linear
`requirements → design → implementation`, then `test` and `docs` run **in parallel**
(both depend only on `implementation`), converging at `compliance`, then a
human-approval-gated `release_readiness`.

**What actually happened, including a real failure loop**: the first run attempt
safe-stopped with a rollback — the `test` stage's `mvnw.cmd` invocation failed under
`subprocess`'s Windows shell resolution (a real bug, not a demo). The rollback was a
harmless no-op (repo was already at that commit), but it also **discarded an
uncommitted fix I had just made to the orchestrator's own `test.py`**, because
`git reset --hard` resets the whole working tree, not just the product code being
gated — a genuine limitation, now documented, with the fix being to commit orchestrator
changes immediately rather than leaving them uncommitted across a run. The second
fixed attempt hit a second real issue (`JAVA_HOME` not set for the subprocess
environment), fixed by adding JDK auto-discovery to the `test` stage runner. The third
attempt hit the `compliance` guardrail correctly blocking `pom.xml` (a protected file)
since the ZXing dependency was added — resolved by explicitly configuring this
pipeline's `compliance` stage to allow `pom.xml`, with the rationale (already-approved
dependency addition) recorded directly in the pipeline spec. The fourth attempt
completed cleanly through to a human-approved release.

**Validation**: real `mvnw test` run (`QrCodeServiceTest`, `ShortUrlControllerTest`),
real `git diff` checks confirming `docs/BACKEND.md`, `docs/schemas.md`, `openapi.yml`
were actually updated, real compliance checks, human approval recorded in the audit
log with approver name and rationale. Final run: 7/7 stages passed, 0 retries, 0
rollbacks (the earlier rollback happened in an attempt before the bugs were fixed, not
in this final recorded run — see `docs/TESTING_AND_LIMITATIONS.md` for why that
matters).

## 2. Brownfield — CORS Configuration

**Input**: "Add CORS configuration to the backend" — a well-defined enhancement to an
*existing*, already-documented gap (`docs/BACKEND.md` and
`docs/FRONTEND_INTEGRATION.md` both flagged "no CORS configuration exists yet" before
this change).

**Codebase reasoning** (`orchestrator/artifacts/brownfield/requirements.md`):
explicitly enumerates the impacted areas before touching anything — no existing
`WebMvcConfigurer` bean (net-new, not a modification), a new `application.yml`
property needed (not hardcoded, since dev/prod origins differ), all controller
endpoints affected equally since CORS is cross-cutting (so no controller changes
needed), and two docs files containing now-stale warnings that must be *corrected*,
not just supplemented.

**Design decision, secure by default**: `app.cors.allowed-origins` defaults to empty —
no cross-origin requests allowed until an operator explicitly configures one or more
origins, consistent with this project's existing conservative defaults (strict URL
validation, default expiry).

**Orchestration** (`orchestrator/pipelines/brownfield.json`): same shape as
greenfield (parallel `test`+`docs`, converging at `compliance`, approval-gated
release). This run completed cleanly on the first attempt — 7/7 stages, 0 retries, 0
rollbacks — since the bugs found and fixed during the greenfield run were already
resolved in the engine by this point.

**Validation**: real MockMvc tests sending actual `Origin`/preflight headers and
asserting the real Spring CORS filter's response (`Access-Control-Allow-Origin`
present for a configured origin, `403` for an unconfigured one, `403` when nothing is
configured at all) — not a test of "was the config bean registered," a test of
whether cross-origin requests are actually allowed or blocked.

## 3. Ambiguous — "Add Analytics"

**Input**: "Add analytics to the URL shortener" — deliberately vague. Doesn't specify
what to measure, how it's surfaced, granularity, or who can see it.

**Why this is the most important scenario**: there's no human available synchronously
inside an automated pipeline stage to resolve ambiguity. The requirements stage's
runner (`orchestrator/stages/requirements.py`) enforces this structurally — when a
stage's params set `"ambiguous": true`, the artifact **must** contain an
`## Assumptions` section or the stage fails its own exit gate, regardless of approval.
See `orchestrator/artifacts/ambiguous/requirements.md` for the full ambiguity writeup:
what's unclear, why, and seven explicit numbered assumptions (daily counts + top
referrers only; no unique-visitor/geo/device tracking; a new dedicated endpoint, not a
modified existing one; no new UI; same no-auth access model as the rest of the API; no
historical backfill).

**The key differentiator**: `orchestrator/pipelines/ambiguous.json` marks
`requires_approval: true` on the **requirements** stage itself, not just on
`release_readiness`. The run genuinely halted twice — once immediately after
`requirements` (before design or implementation had even started), and again before
release. This demonstrates the "human approval checkpoints for high-impact actions"
requirement applied to the specific action that matters most here: signing off on
*which interpretation of a vague requirement is correct*, before any engineering
effort is spent building it.

**What was approved and why** (recorded in the real audit log, not paraphrased here):
the interpretation was reviewed against the project's existing no-auth model and
approved as "a reasonable, appropriately-scoped reading of 'add analytics'."

**Implementation**: a new `click_events` table (one row per redirect: timestamp,
referrer), a new `AnalyticsService` that aggregates in **Java** rather than SQL
specifically so the grouping/sorting/top-N logic is unit-testable without a database
(see `AnalyticsServiceTest` — `aggregateByDay`/`aggregateByReferrer` are tested as pure
functions), and a new `GET /api/urls/{shortCode}/analytics` endpoint.

**Orchestration**: `requirements` (approval-gated) → `design` → `implementation` →
parallel `test`+`docs` → `compliance` → `release_readiness` (approval-gated). Final
run: 7/7 stages passed, 0 retries, 0 rollbacks, 2 human approvals recorded.
